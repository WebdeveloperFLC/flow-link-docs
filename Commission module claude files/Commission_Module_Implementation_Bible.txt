0. Purpose, Scope & How Cursor Must Use This Document
This is the Implementation Bible for the Future Link ERP Commission Module. It converts the approved TO-BE architecture into a build specification. It tells Cursor exactly what to implement, in what order, and how to prove it works. Where the earlier Review said “what should change,” this document says “how to build it and when it is done.”
0.1 Scope decision (binding)
Future Link operates many service lines — institution recruitment, visa and immigration services, coaching, English/French/German language, consultation, government-fee handling, internal referrals, franchise branches, partner offices, and counselors. The Commission Module will not implement all of these now. Instead it builds a service-line-agnostic payout core and implements the institution-commission service line first. Every other service line is a future phase that plugs into the same core.
Design-for-many, build-one:  The payee, earning, rule, and event models MUST be built generically enough that adding “visa service payout” or “referral payout” later is configuration + a new rule set, not a schema redesign. But only the institution-commission service line is implemented and tested in Phases 3–6. Do not build visa/coaching/referral logic yet — only leave the seams.

Every feature in §8 uses this exact structure. Cursor must not implement a feature that is missing any field.

0.3 REUSE → EXTEND → CREATE decision rule
Before writing any code for a capability, Cursor applies this test in order and records the outcome in the PR description:
	•	REUSE — does a cited existing symbol already do this? If yes, call it. Never duplicate simulateCommission, resolveCommissionForStudent, the Phase 1 lifecycle RPCs, or the receipt state functions.
	•	EXTEND — can an existing table/RPC/component be widened with additive columns or parameters? Prefer this over new entities.
	•	CREATE — only when neither applies. New entities must follow §3 standards and publish events per §5.

1. Single Source of Truth (SSOT) Principles
These principles are the constitution of the module. Every later section derives from them.
#
Principle
Consequence for implementation
1
The database is the source of truth, not the UI.
No business rule lives only in React. Every rule that guards money exists in SQL (RPC/constraint/trigger).
2
All business writes go through RPCs.
No direct .insert/.update on financial tables from the client. Preserve the existing RPC-mediated write model.
3
Calculation is deterministic and reproducible.
Same inputs → same output. simulateCommission (TS) and fn_resolve_commission_rule (SQL) MUST stay mirrored.
4
Finance is the SSOT for accounting.
The Commission Module never owns journals, ledgers, tax, or period close. It publishes events; Finance posts them (§5).
5
Immutable records are never edited, only superseded.
Snapshots and posted receipts are append-only. Corrections are new adjustment/credit/debit events.
6
Every financial mutation is audited.
Actor, timestamp, before/after, reason recorded in an append-only audit log (§8, F3.3).
7
Configuration lives in data, not code.
Thresholds, rounding, FX fallbacks, and channel toggles are rows, not constants.
8
One earning may have many payees.
The payee/earning-allocation model is the core seam for all future service lines and split/multi-level payouts.

3. Database Standards & Conventions
All new tables and columns follow these conventions so the schema stays consistent with the existing upi_commission* namespace.
3.1 Naming & structure
	•	Namespace: all commission tables are prefixed upi_commission_* (or upi_* for shared masters). New payee/earning tables follow the same prefix.
	•	Primary keys: id uuid default gen_random_uuid(). Masters keyed on a text code where an existing pattern exists (hold reasons, periods).
	•	Money & rates: amounts numeric(14,2) with CHECK (>= 0) or (> 0) as appropriate; exchange rates numeric(12,6) CHECK (> 0).
	•	Timestamps: created_at, updated_at timestamptz default now(); updated_at maintained by a touch_updated_at() trigger (existing pattern).
	•	Soft delete: financial tables carry deleted_at timestamptz null; all reads filter deleted_at IS NULL (NN-10).
	•	Currency columns: every monetary row records its own currency (e.g. earning_currency, payout_currency) — never assume a global currency (§4).
3.2 Mandatory columns on every NEW financial table
id            uuid       primary key default gen_random_uuid() created_at    timestamptz not null default now() created_by    uuid       not null            -- actor (auth.uid()) updated_at    timestamptz not null default now() deleted_at    timestamptz                    -- soft delete (NN-10) -- money columns: numeric(14,2); rates numeric(12,6) -- currency columns: text not null (ISO 4217)
3.3 Integrity & audit
	•	Every mutating RPC writes one audit row (actor, entity, entity_id, action, before jsonb, after jsonb, reason) before returning.
	•	Foreign keys are declared and indexed. No orphan financial rows.
	•	Immutability is enforced by trigger, not by application convention (follow trg_block_receipt_edit / trg_block_snapshot_update).
	•	RLS: every new financial table gets a role/entity-scoped policy. Never ship an auth_all USING(true) policy (that pattern is being remediated, not repeated).
3.4 Additive-only migrations
Migrations are forward-only and additive. Do not drop or rename existing commission columns/tables. New capability is delivered by adding columns/tables and back-filling, so Phase 1–2B behavior is never disturbed (regression safety, §14).

4.2 FX rules (binding)
	•	Locked rates: the exchange_rate on a receipt/earning is captured and frozen at the moment of the financial event. base_amount is derived and never re-derived retroactively (NN-9).
	•	Effective-dated history: create an fx_rate_history table (effective_from, from_ccy, to_ccy, rate, source). Replace the hardcoded CAD=61.7 fallback with a looked-up, dated rate. CREATE.
	•	FX gain/loss: when settlement currency differs from earning currency, the realized difference is computed and published to Finance as an FX gain/loss event — the Commission Module does not book it itself (§5).
	•	FX review gate: mandatory when receipt currency ≠ invoice currency (NN-4) — reuse needsFxReview and fn_approve_receipt_fx_review.
	•	No silent conversion: conversions always record from_ccy, to_ccy, rate, and rate source. A converted amount without a recorded rate is a defect.
Reuse note:  The receipts table already implements the locked-rate pattern correctly: exchange_rate numeric(12,6) plus base_amount GENERATED ALWAYS AS (round(receipt_amount * exchange_rate, 2)) STORED. New earning/payout tables MUST copy this exact pattern rather than invent a new one.

5.2 The financial-event contract
The reserved accounting_journal_id column is the seam. Instead of populating a journal directly, the posting RPC writes a commission_financial_events row (append-only) describing what happened, and Finance links back its journal id. The Commission Module treats accounting_journal_id as written-by-Finance, read-only to Commission.
commission_financial_events (append-only)   id, event_type, occurred_at, entity_type, entity_id,   amount numeric(14,2), currency text, base_amount numeric(14,2), base_currency text,   payee_id, institution_id, legal_entity_id,   status: pending | posted | rejected,   finance_journal_id uuid,   -- set by Finance, not by Commission   created_by, created_at   event_type ∈ { receipt_posted, receipt_voided, commission_earned,                adjustment_raised, credit_note_issued, debit_note_issued,                write_off_approved, fx_gain_loss, clawback_applied }
5.3 Event lifecycle
	•	A mutating Commission RPC (e.g. fn_post_commission_receipt) inserts a commission_financial_event with status=pending inside the same transaction.
	•	Finance consumes pending events, posts journals, sets finance_journal_id and status=posted, and (only Finance) writes back accounting_journal_id on the source row.
	•	If Finance rejects (status=rejected), Commission surfaces it as a reconciliation exception — it does not retry-post itself.
	•	Commission never reads or writes journal/ledger tables. It reads only its own events + the finance_journal_id echoed back.
Why this matters:  This keeps Commission and Finance independently evolvable and prevents two modules from disagreeing about the books. It also means every other future service line (visa, coaching, referral) posts to Finance through the SAME event contract — one integration, many producers.

6.2 Design constraints
	•	The rules engine is service-line-agnostic: a rule references an earning context (attributes) and produces a payout, regardless of whether the earning is an institution commission or a future visa-service fee.
	•	Rules are data. No payout condition is hardcoded in TS/SQL beyond the generic operators already supported (=, !=, >, <, in) plus effective-date and scope matching.
	•	Draft → (approve) → published → (supersede) is the only lifecycle. Published versions are immutable (consistent with NN-2 philosophy).
	•	Resolution remains deterministic and mirrored between TS (preview) and SQL (authoritative).

7. Core Data Model — Extensible Payee & Earning Model
This is the central EXTEND of the entire program and the seam that unlocks split, shared, multi-level, agent, counselor, branch, and future service-line payouts. Build it in Phase 5, but design every earlier phase to be compatible with it.
7.1 Concept
Today a upi_commission_students row conflates “the earning” and “the single payee.” The Bible separates them: an earning (money generated by a student/service event) can be allocated to one or many payees via payee-allocation rows.
commission_payees            -- WHO gets paid (institution, aggregator, agent,                                 counselor, branch, referral source, …)   id, payee_type, display_name, legal_entity_id, default_currency, …   commission_earnings          -- WHAT was earned (generalizes the student row)   id, source_type, source_id (e.g. student), earning_currency,   gross_amount, matched_rule_version_id, snapshot_id, …   commission_earning_allocations   -- HOW it splits across payees   id, earning_id, payee_id, allocation_type (percent|fixed),   allocation_value, payout_amount, payout_currency, level (for multi-level), …
7.2 Migration strategy (non-disruptive)
	•	Introduce the three tables additively. Do NOT drop upi_commission_students.
	•	Back-fill: for every existing student row, create one earning + one 100%-to-institution allocation. This preserves current behavior exactly (a single-payee earning is just the degenerate case).
	•	Point new reads at the earning/allocation model; keep the legacy student view synced via the existing trg_sync_ucs_legacy_status pattern so Phase 1–2B UI keeps working.
	•	Only after parity is proven (regression suite green) do split/multi-level features activate.
Compatibility guarantee:  Because a single-payee earning is the degenerate case of the general model, the institution-commission service line behaves identically before and after this EXTEND. That is the acceptance bar for Phase 5’s first milestone (§13).

8. Feature Specifications (Phase 3–6)
Each feature below is self-contained and follows the §0.2 format. Implement in the §12 order. IDs map to the roadmap phase (F3.x = Phase 3, etc.).
8.1 Phase 3 — Financial Backbone & Controls
F3.1  Financial-event publishing (Commission → Finance)
Purpose
Establish the SSOT boundary: Commission emits financial events; Finance posts journals. Rationale: keeps books consistent and lets all future service lines share one Finance integration (§5).
Business Rules
	•	NN-5: Commission posts no journals itself.
	•	Every money-moving RPC emits exactly one event in-transaction.
	•	accounting_journal_id is written only by Finance; Commission treats it read-only.
Implementation Sequence
	•	Migration: create commission_financial_events (append-only, §5.2 shape).
	•	EXTEND fn_post_commission_receipt / void to insert a pending event.
	•	Add a read API/view for reconciliation status.
	•	No UI change required; expose event status on the receipt detail.
Acceptance Criteria
	•	Posting a receipt creates one pending financial event with correct amount/currency/base_amount.
	•	Voiding creates a compensating event.
	•	accounting_journal_id remains NULL until Finance writes it (matches UAT 2A-18 until Finance is live).
Regression Requirements
	•	Existing post/void/allocation behavior unchanged (Phase 2A UAT still passes).
	•	trg_block_receipt_edit still blocks edits on posted/voided receipts.

F3.2  Adjustments, credit notes & debit notes
Purpose
Provide immutable-safe corrections. Rationale: snapshots/receipts are immutable (NN-1/2), so corrections must be additive posting events, never edits.
Business Rules
	•	A correction never edits the original row; it creates a linked adjustment/credit/debit record.
	•	Each correction emits its own financial event (§5).
	•	Credit/debit notes reference the invoice they correct.
Implementation Sequence
	•	Migration: commission_adjustments, commission_credit_notes, commission_debit_notes.
	•	RPCs to raise each, with approval gate hook (F4.1).
	•	Emit financial events; link to original entity.
	•	UI: correction action on invoice/receipt detail (post-approval).
Acceptance Criteria
	•	Raising a credit note leaves the original invoice unchanged and produces a linked note + event.
	•	Net outstanding reflects the correction.
Regression Requirements
	•	Original invoice/receipt totals and immutability unchanged.

F3.3  Append-only audit log
Purpose
Defensible who-changed-what trail on financial mutations. Rationale: NN-6; required for audit/compliance and the §9 audit reports.
Business Rules
	•	Every mutating financial RPC writes one audit row before returning.
	•	Audit rows capture actor, entity, action, before/after (jsonb), reason.
	•	Audit log is append-only (no update/delete) — enforce by trigger.
Implementation Sequence
	•	Migration: commission_audit_log + block-update/delete trigger.
	•	Add audit insert to each mutating RPC (receipt, invoice, adjustment, hold, transfer, rule publish).
	•	UI: read-only audit timeline on entity detail.
Acceptance Criteria
	•	Every financial state change produces exactly one audit row with correct before/after.
	•	Audit rows cannot be updated or deleted.
Regression Requirements
	•	Mutating RPCs return same results; only an audit row is added.

F3.4  RLS remediation on financial tables
Purpose
Close the top security gap. Rationale: existing auth_all USING(true) policies let any authenticated user mutate financial rows (Review §8, Critical).
Business Rules
	•	Replace auth_all policies on upi_commission_students / _invoices / _invoice_line_items with role- and entity-scoped policies.
	•	No new table ships with USING(true).
Implementation Sequence
	•	Migration: drop permissive policies; add scoped policies (commission admin / accounting / entity).
	•	Verify counselor confidential view still hides amounts.
Acceptance Criteria
	•	A non-privileged authenticated user can no longer read/write financial rows directly.
	•	Existing privileged workflows unaffected.
Regression Requirements
	•	All Phase 1–2B UATs pass under the tightened policies.

8.2 Phase 4 — Approvals, Audit & Automation
F4.1  Workflow & approval engine (maker-checker)
Purpose
Gate financial actions with multi-eyes approval. Rationale: NN-7; segregation of duties is impossible without it.
Business Rules
	•	Actions above a configurable threshold require a different approver than the maker.
	•	Approval state is persisted, not implicit.
	•	Approve/reject writes an audit row (F3.3).
Implementation Sequence
	•	Migration: commission_workflows, commission_approvals, commission_approval_steps.
	•	RPC gate: post-receipt, generate-invoice, write-off, adjustment, rule-publish route through approval.
	•	UI: approval queue + approve/reject with reason.
Acceptance Criteria
	•	A maker cannot approve their own above-threshold action.
	•	Below-threshold actions proceed without approval (configurable).
Regression Requirements
	•	Below-threshold flows behave as today; no silent gating of existing low-risk actions.

F4.2  Activate CRM/Admission auto student-row (earning) creation
Purpose
Remove the manual data-entry bottleneck. Rationale: clientIntegrationBridge.ts is designed but dormant (BRIDGE_ENABLED=false); auto-creation is the top scale bottleneck.
Business Rules
	•	An admission/CRM event (e.g. application_submitted, deposit paid) creates a pending earning.
	•	De-duplicate against the existing eligibility evaluator — do NOT introduce a third eligibility code path.
Implementation Sequence
	•	Consolidate eligibility logic: the dormant bridge checker must call the SAME rule as fn_evaluate_eligibility (remove the duplicate).
	•	Set BRIDGE_ENABLED true behind a config flag; wire event listeners.
	•	Emit suggestions; create pending earnings; never auto-claim.
Acceptance Criteria
	•	A qualifying admission event creates exactly one pending earning.
	•	Eligibility outcome matches the single canonical evaluator.
Regression Requirements
	•	Manual creation still works; no double-creation when both manual and event paths fire.

F4.3  Notifications & email dispatch
Purpose
Drive approval, dunning, dispute, and renewal alerts. Rationale: workflows in F4.1 depend on notifying approvers; no commission notifications exist today.
Business Rules
	•	Every notification is recorded in a notification log (auditable).
	•	Templates are configuration, not code.
Implementation Sequence
	•	Migration: commission_notifications + log.
	•	Hook approval-pending, claim-overdue, dispute-raised, agreement-renewal events.
	•	Dispatch via the platform notification/email service.
Acceptance Criteria
	•	An approval-pending action notifies an eligible approver and logs it.
Regression Requirements
	•	No workflow blocks if notification dispatch fails (log + continue).

F4.4  Soft deletes on financial tables
Purpose
Recoverability + audit. Rationale: NN-10; financial rows must never be physically destroyed.
Business Rules
	•	deleted_at set instead of DELETE; all reads filter deleted_at IS NULL.
	•	Immutable rows (posted receipts, snapshots) still cannot be soft-deleted either.
Implementation Sequence
	•	Migration: add deleted_at to financial tables; update read views/RPCs to filter.
	•	Replace any hard-delete paths with soft-delete RPCs.
Acceptance Criteria
	•	“Deleting” a draft hides it from lists but retains the row + audit trail.
Regression Requirements
	•	Existing lists show identical active rows (deleted_at IS NULL default).

8.3 Phase 5 — Multi-Entity & Multi-Channel
F5.1  Payee + earning-allocation model
Purpose
The core EXTEND (§7). Unlocks split, shared, multi-level, agent, counselor, branch, and future service-line payouts.
Business Rules
	•	One earning → one or many payee allocations; allocations sum to 100% (percent) or to gross (fixed).
	•	Single-payee institution commission is the degenerate case and must behave identically to today.
	•	Each allocation records its own payout currency.
Implementation Sequence
	•	Migrations: commission_payees, commission_earnings, commission_earning_allocations (§7.1).
	•	Back-fill one earning + one 100% institution allocation per existing student row.
	•	Sync legacy student view; prove parity before enabling splits.
	•	Then enable split/multi-level allocation UI + RPCs.
Acceptance Criteria
	•	Post back-fill, all Phase 1–2B numbers are byte-identical (parity milestone).
	•	A two-payee split produces two allocations summing to the earning.
Regression Requirements
	•	Full Phase 1–2B UAT suite passes unchanged after back-fill.

F5.2  Org-unit / legal-entity / company dimensions
Purpose
Enable multi-company, multi-brand, multi-branch, multi-country scale. Rationale: Review §11 identifies these as the true enterprise-scale blockers.
Business Rules
	•	Every earning/allocation carries legal_entity_id and (optional) org_unit_id / brand_id.
	•	Base currency and tax jurisdiction derive from the legal entity.
Implementation Sequence
	•	Migrations: org-unit / legal-entity / brand dimension tables + FKs.
	•	Back-fill existing rows to the default entity.
	•	Scope reads/reports by dimension.
Acceptance Criteria
	•	Commission can be reported and settled per legal entity and currency.
Regression Requirements
	•	Single-entity deployments behave exactly as today (default entity).

F5.3  Tax/GST engine + effective-dated FX + clawback
Purpose
Complete the financial correctness set. Rationale: tax computation, dated FX, and clawback are all Missing/partial (Review §5, §4).
Business Rules
	•	Tax computed per jurisdiction from config; results published to Finance (Finance files).
	•	fx_rate_history replaces the hardcoded fallback; conversions record the dated rate used.
	•	Clawback uses the reserved clawback_amount/clawback_status columns and emits a clawback event.
Implementation Sequence
	•	Migrations: tax config, fx_rate_history; wire clawback RPC to existing columns.
	•	Compute tax on invoice/receipt; emit events; never book locally.
Acceptance Criteria
	•	A GST-applicable invoice shows tax; a clawback reduces payout and emits an event.
Regression Requirements
	•	Zero-tax jurisdictions and CAD-only flows behave as today.

8.4 Phase 6 — Reporting, Recognition & Close
F6.1  Reporting mart + core report suite
Purpose
Deliver enterprise reporting without straining transactional tables. Rationale: no materialized views/reports exist today (Review §9).
Business Rules
	•	Reports read from materialized views / aggregate tables, refreshed on a schedule — not from live transactional joins at scale.
	•	Every report respects RLS and entity scope.
Implementation Sequence
	•	Create materialized views: aging, outstanding, pipeline, settlement, profitability.
	•	Scheduled refresh job.
	•	UI: report screens with drill-down + export (reuse claimsExport patterns).
Acceptance Criteria
	•	Aging and pipeline reports match transactional truth at a chosen cut date.
Regression Requirements
	•	Existing KPI strips continue to function; reports are additive.

F6.2  Reconciliation, period close & revenue recognition (Finance-led)
Purpose
Close the finance-operations loop. Rationale: three-way reconciliation, period close, and recognition are Missing (Review §5).
Business Rules
	•	Commission PROVIDES data; Finance OWNS close, lock, and recognition (§5).
	•	Once a period is closed by Finance, Commission blocks back-dated financial events into it.
Implementation Sequence
	•	Expose reconciliation data (invoice–receipt–event) to Finance.
	•	Honor a Finance period-lock signal; reject events into closed periods (surface as exception).
Acceptance Criteria

16. Rollback Strategy
Because migrations are additive and behavior changes are gated, each step is independently reversible with minimal blast radius.
Layer
Rollback approach
Feature flags
Every new behavior (auto-creation, approvals, splits) ships behind a config flag. Rollback = flip the flag off; data model stays.
Additive migrations
New tables/columns are additive; disabling a feature leaves them dormant and harmless (mirrors the existing dormant-bridge pattern).
Event publishing
If Finance integration must be paused, events queue as pending; nothing double-posts because Commission never posts journals itself.
Payee/earning back-fill
Legacy student rows are retained and kept in sync; revert = read from the legacy view until parity is re-confirmed.
Approvals
Threshold can be raised to effectively disable gating without code change.
RLS
Policies are versioned migrations; a scoped policy can be replaced by a prior migration if an access regression appears (then fixed forward — never revert to auth_all).
Data safety
Soft deletes + append-only audit mean no rollback destroys financial history.

Rollback principle:  No rollback ever edits or deletes an immutable or financial record. Reversal is always additive (compensating event, flag flip, or superseding version) — the same discipline the module already applies to receipts and snapshots.
