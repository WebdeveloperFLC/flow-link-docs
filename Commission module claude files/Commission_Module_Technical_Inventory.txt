1. Executive Summary
The Commission module automates institution partner commissions for an international education agency: from the commercial agreement with a partner institution, through per-student commission calculation and eligibility, into claims, invoicing, receipt posting, and aggregator reconciliation. It is a Supabase-first application — persistence, business logic, access control, and workflow all run through PostgreSQL tables, RPC functions, row-level security, and Edge Functions, with a React institution-tab workspace as the operational core.
Overall completion is estimated at ~72% for institution partner-commission operations through Phase 2B. The three delivered phases (Phase 1 lifecycle, Phase 2A receipts, Phase 2B aggregator) are internally documented as UAT-ready. Enterprise integrations — Finance general-ledger posting, CRM automation, and additional commission channels (referral, B2B, agent) — remain largely unbuilt and are documented as out-of-scope or missing.

2.7 Indexes, storage buckets & RLS
Indexes (selected): idx_upi_billing_profiles_inst and a unique partial index for default billing profiles; idx_ucs_cycle, idx_ucs_inst, idx_ucs_status, idx_ucs_lifecycle, idx_ucs_period on students; a unique (client_id, partnership_route_id, commission_period_code) where client_id NOT NULL; receipt indexes on status, payer, and remittance reference.
Storage buckets: upi-commission-receipts (receipt attachments) and upi-commission-aggregator-statements (aggregator batch statements).
Row-level security: Confidential tables (billing profiles, eligibility configs, transfer events, snapshots) use a can_view_upi_confidential(auth.uid()) policy. Hold reasons and periods are readable by all authenticated users. An early upi_commission_students policy remains permissive (authenticated USING true) — flagged in Technical Debt.
2.8 Existing data flow
The end-to-end persistence flow moves from institution setup, through per-student calculation, into claims and invoicing, then receipts, and finally aggregator reconciliation:
	•	Setup: billing profiles, agreement versions, eligibility configs, and commissions+rules are authored per institution; partnership routes carry a default_commission_id.
	•	Per-student: upi_commission_students rows are calculated against a published commission, producing an immutable upi_commission_snapshots record; transfer events branch replacement rows.
	•	Claims: students are grouped into upi_claim_cycles; submitting a cycle generates upi_commission_invoices with upi_invoice_line_items.
	•	Receipts (2A): upi_commission_receipts are allocated to invoices and students; posting updates payment_status columns.
	•	Aggregator (2B): remittance_batches and aggregator_invoices group receipts and invoices for reconciliation.

6. Existing Business Logic
Business rules are implemented twice by design: pure TypeScript functions power the client-side simulator and previews, while SQL RPCs are the authoritative source of truth for persisted writes. The two are intended to mirror each other.
6.1 Rule resolution precedence
Evidence: commissionRuleResolver.ts and fn_resolve_commission_rule. Rules are matched most-specific-first:
Promotion → Intake → Program → Category → Campus → Country → Default.
Commission selection resolves the partnership route’s default_commission_id first; if absent, it falls back to the institution’s active commission by effective_from.
6.2 Commission calculation
Evidence: commissionEngine.ts.
	•	Base: tuition × (base_rate_percent / 100).
	•	Rule payout types: fixed, percentage, multiplier.
	•	Slab tiers match on student_count against min_value / max_value.
	•	Conditional rules match condition_field/operator/value using =, !=, >, <, in.
6.3 Eligibility
Evidence: commissionEligibilityEvaluator.ts, fn_evaluate_eligibility.

6.4 Claim readiness
Evidence: CommissionLifecycleDialog.tsx — isReadyForClaim() requires eligible + claim ready + no active hold, checked before submit and invoice.
6.5 Transfer outcomes
Evidence: commissionTransferHelpers.ts, fn_process_transfer_outcome. Outcomes are unchanged, amended, cancelled, replaced, under_review; a replacement creates a new student row and a new snapshot path.
6.6 Receipt state machine
Evidence: commissionReceiptRules.ts.
draft → ready → posted                  ↓ voided draft/ready editable; posted/voided blocked by DB trigger
Posting requires full allocation; FX review is required when the receipt currency differs from the invoice currency.
6.7 Currency handling
	•	Per-student columns: snapshot_currency, invoice_currency, receipt_currency, base_currency.
	•	Receipts carry exchange_rate and a generated base_amount.
	•	Performance Hub converts to INR via fx_rates.rate_to_inr (fallback CAD = 61.7).
6.8 Finance posting
Not Found. Receipt posting updates student and invoice payment columns; the accounting_journal_id column on receipts is not populated by any observed code.

14. Deliverables Summary & Handoff Notes
This document is the implementation baseline for the Commission module in flow-link-docs. It covers all fourteen audit areas: executive summary, database, business process, backend, frontend, business logic, API inventory, component inventory, UI (now with captures), technical debt, reuse analysis, extension points, and cross-module dependencies.
14.1 Key takeaways for an incoming architect
	•	Terminology: “Commission Plan” = upi_commissions; there is no separate plan entity.
	•	Architecture: Supabase RPC-centric backend; the React institution-tab workspace is the operational core; all writes go through SECURITY DEFINER RPCs.
	•	Phases delivered: Phase 1 (lifecycle), 2A (receipts), 2B (aggregator) — evidenced by migrations 20260723120000 through 20260815120500 and the corresponding UI.
	•	Explicit gaps: Finance GL bridge, CRM automation, custom eligibility, claim eligibility rules, global admin/reports/pending-claims screens, and referral/B2B/agent commission types.
	•	Reuse principle: Preserve commissionEngine, the rule resolver, Phase 1 lifecycle RPCs, the receipt allocation model, and the institution-tab structure; extend rather than replace.
